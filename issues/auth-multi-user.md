> - Ports: API `:4000`, web `:5200`, Postgres `:5433`. No `Co-Authored-By` trailer in commits.

**Type:** AFK ¬∑ **Blocked by:** none ¬∑ **Status: ‚úÖ Done** ‚Äî better-auth 1.6.19 mounted at

## What to build

The thinnest complete identity path through every layer, **without yet gating the rest of the API** ‚Äî
so the app is never left locked before it has a door. Stand up `better-auth` in the API: install it,
mount its request handler on the existing `NestExpressApplication` at **`/api/auth/*`**, point it at
the same Postgres pool, and configure **email/password** + httpOnly cookie sessions. Generate
better-auth's tables (`user`, `session`, `account`, `verification`) with its schema CLI and **paste the
`CREATE TABLE ‚Ä¶ IF NOT EXISTS` DDL into the bootstrap `DDL`** (`@varys/db`), additive and idempotent.
Read config from env (`BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`) and document it in the README +
`docker-compose`. Add **`/api/auth` to the Vite dev proxy allowlist**.

On the web side, add a **login screen** (email/password sign-up + sign-in) built on `@varys/ui`, a
`useSession()`/`whoami` bootstrap that shows a splash while unknown and the login screen when
unauthenticated, and a **user menu + sign-out in `TopBar`**. The app still renders for everyone (no
guard yet), but a signed-in user sees their identity and can sign out ‚Äî the round-trip is real
end-to-end.

## Acceptance criteria

- [ ] `better-auth` is mounted at `/api/auth/*` on the existing Express app, backed by the same Postgres pool; email/password sign-up, sign-in, session, and sign-out all work via `/api/auth/*`.
- [ ] Sessions are **httpOnly cookies** (not JWT-in-localStorage); the cookie is set on sign-in and cleared on sign-out.
- [ ] better-auth's tables are added to the bootstrap `DDL` (`IF NOT EXISTS`, additive); **restart `pnpm dev`** applies them; existing tables/data are untouched.
- [ ] `BETTER_AUTH_SECRET` + `BETTER_AUTH_URL` are read from env (not committed) and documented in the README run guide + `docker-compose`.
- [ ] `/api/auth` is in the Vite dev proxy allowlist; the login round-trip works through the `:5200` proxy (no "API returned HTML").
- [ ] Web: a login screen (email/password) on `@varys/ui`; `useSession()`/`whoami` bootstrap (splash while unknown ‚Üí login when signed out ‚Üí app when signed in); user menu + sign-out in `TopBar` showing the signed-in identity.
- [ ] New providers (auth config provider) use explicit `@Inject(...)`; the API dev server boots.
- [ ] `pnpm --filter @varys/web build` is clean; existing `runs.e2e` / `tests.e2e` still pass (no guard yet ‚Üí no harness change needed in this issue).

## What to build

Flip the API to **deny-by-default** and make every client cope. Add a **global `APP_GUARD`** that
resolves the better-auth session from the request cookie, attaches the resolved `user` to the request,
and returns **401** when there's no valid session. **Allowlist** exactly: `/api/auth/*`,
`/trace-viewer/*` (self-hosted static assets), a new public **`GET /health`**, and **`/mcp`** (this
slice only, per the locked decision). Add `/health` to the Vite proxy allowlist.

On the web, treat a **401 as "session gone" ‚Üí redirect to the login screen** in the fetch/TanStack
Query layer (no toast spam, no half-rendered data). The extension reuses the **browser session
cookie**: add `credentials: 'include'` to its API calls and enable **CORS with `credentials: true` for
the extension origin** on the API. **Verify the cookie actually reaches the API from the extension**;
if SameSite blocks the cross-origin cookie, implement the **`chrome.cookies` host-permission fallback**
(read the API-origin session cookie and attach it explicitly). Finally, **fix the E2E harness** so the
existing API E2Es sign in once and reuse the session cookie (exercising the real auth path), since they
now hit gated routes.

## Acceptance criteria

- [ ] A global `APP_GUARD` resolves the session from the cookie, attaches `request.user`, and returns **401** for any non-allowlisted route without a valid session; the guard uses explicit `@Inject(...)` and the dev server boots.
- [ ] Allowlist is exactly `/api/auth/*`, `/trace-viewer/*`, `GET /health`, and `/mcp`; `/health` returns a liveness payload unauthenticated and is added to the Vite proxy; `/mcp` remains reachable unauthenticated (documented accepted risk).
- [ ] Web: a 401 from any API call redirects to the login screen (session-expiry path), and a signed-in user reaches all data exactly as before.
- [ ] Extension: API calls send `credentials: 'include'`; the API enables CORS with `credentials: true` for the extension origin (not `*`); a recording **saves successfully while the user is logged into the web app**. *(Manual smoke ‚Äî the cross-origin cookie is the slice's known risk; the `chrome.cookies` fallback is implemented if SameSite blocks the cookie.)*
- [ ] The shared API E2E harness signs in and reuses the session cookie; **all existing E2Es pass with the gate on** (no test-only guard bypass).
- [ ] **Chromium-free API E2E** (`apps/api/test/auth.e2e.spec.ts`): unauth `GET /tests` ‚Üí 401; sign-up + sign-in ‚Üí cookie ‚Üí `GET /tests` ‚Üí 200; sign-out ‚Üí 401 again; `/health` + `/mcp` reachable unauthenticated. Prior art: `tests.e2e`.
- [ ] `pnpm --filter @varys/web build` is clean.

## What to build

Add the second sign-in method: **Google SSO with an env-controlled domain restriction**. Configure
better-auth's **Google social provider** (`GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` from env;
documented), add a **"Sign in with Google"** button to the login screen (shown only when `google` is
enabled ‚Äî the `VARYS_AUTH_METHODS` toggle scaffolded in Issue 1), and wire the Google branch of
`resolveAuthMethods` so `VARYS_AUTH_METHODS=‚Ä¶,google` becomes functional. A first valid sign-in
**auto-provisions** the user.

**The domain restriction is env-driven** (per user direction ‚Äî "that is controllable using an env
variable"), not hardcoded:
- `VARYS_AUTH_ALLOWED_DOMAINS` ‚Äî comma list of permitted email domains (default `datagenie.ai`; empty
  = no restriction). A sign-in whose email domain (or Google `hd` hosted-domain claim) is not in the
  list is **rejected** (no user provisioned, clear error).
- `VARYS_AUTH_DOMAIN_SCOPE` ‚Äî `google` (default; restrict only SSO, matching DESIGN ¬ß11) or `all`
  (restrict every sign-up, email/password included). This makes the "Google-only vs all sign-ups"
  question a config flip rather than a code decision.

Pull the "is this email/`hd` allowed given (domains, scope, method)?" check into a **pure predicate** so
it can be unit-tested without a live Google flow. Surface the resolved methods via `GET /auth-config`
(new public route ‚Üí add to the Vite proxy and the Issue 2 guard allowlist) so the login screen renders
only the enabled methods.

## Acceptance criteria

- [ ] better-auth's Google provider is configured from `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` (env, not committed; documented in README); the OAuth callback route is under `/api/auth/*` (already proxied/allowlisted from Issues 1‚Äì2); the `google` branch of `resolveAuthMethods` is now functional.
- [ ] The login screen shows a working "Sign in with Google" button **only when `google` is enabled** (driven by `GET /auth-config`); with `VARYS_AUTH_METHODS=google` the email/password form is hidden, with `password,google` both appear.
- [ ] **Domain restriction is env-driven:** `VARYS_AUTH_ALLOWED_DOMAINS` (default `datagenie.ai`; empty = unrestricted) and `VARYS_AUTH_DOMAIN_SCOPE` (`google` default | `all`). An allowed-domain account signs in (auto-provisioned on first use); a disallowed account is **rejected** (no user created, clear error); with scope `all` the rule also gates email/password sign-up.
- [ ] The allow/deny decision is a **pure predicate with unit tests** (allowed domain accepts; other rejects; scope `google` leaves email/password unrestricted; scope `all` gates it; empty domain list = unrestricted) ‚Äî prior art: `packages/recorder/src/index.spec.ts`.
- [ ] `GET /auth-config` returns the resolved enabled methods, is **public** (in the Vite proxy + the Issue 2 guard allowlist), and the login screen consumes it.
